# Agent-Nutri Pro — Главный маршрутизатор

Перед любой командой сначала прочитай RULES.md.

**Иерархия источников правды → `v2/HIERARCHY.md`** (читать при любом конфликте правил).

RULES.md — общие правила. При конфликте с `v2/` — побеждает `v2/`.
AGENT.md отвечает за маршрутизацию команд и последовательность выполнения.
skills/*.md содержат только специализированную логику конкретного шага.

## Роль

Агент выполняет клинический нутрициологический разбор по данным клиента и базе проекта.

## ПРАВИЛО ВВОДА ДАННЫХ

Пользователь может передавать дополнительные данные прямо в сообщении с командой:

/consult Petrov
Рост: 175 см
Вес: 82 кг
Возраст: 45 лет
Пол: М
Жалобы: усталость, вздутие после еды, тяга к сладкому
Лекарства: метформин 500 мг 2р/день
Диагнозы: СД2, НАЖБП
Активность: офис, без спорта
Курение: нет
Алкоголь: редко

Агент обязан:
1. Извлечь ВСЕ данные из сообщения
2. Сохранить в clients/[Фамилия]/profile.md (дополнить, не затереть)
3. Использовать при анализе
4. Если нет роста или веса — КБЖУ не рассчитать, запросить обязательно
5. Остальное — работать с тем что есть, помечая что не указано

## Пайплайн по умолчанию

1. Прочитать RULES.md.
2. Прочитать входные данные клиента из clients/[Фамилия]/.
3. При необходимости пересчитать единицы по skills/unit_conversion.md.
4. Выполнить профильный skill.
5. Сохранить результат в файл.
6. Обновить learning/cases.md при консультационных командах.
7. Сделать commit и push.
8. Вывести в чат полный текст итогового документа.

## Команды

### /consult [Фамилия]

Полная экспертная консультация. Пайплайн мышления (7 шагов): `v2/PIPELINES.md`.

Два режима:
- **`strict`** (по умолчанию) — не запускать без обязательных данных.
- **`fallback`** — при явной команде «работай с тем что есть»; каждый вывод без данных помечается «⚠».

Порядок:
1. `v2/HIERARCHY.md` + `learning/corrections.md` + `learning/patterns.md`
2. RULES.md
3. Проверка обязательных данных клиента в profile.md: пол, возраст, рост, вес, жалобы, физическая активность, текущие лекарства, хронические заболевания/диагнозы, курение, алкоголь, менструальный цикл (для женщин). **Режим: `strict`.**
4. Если в сообщении к /consult переданы inline-данные, извлечь их и ДОПОЛНИТЬ clients/[Фамилия]/profile.md до запуска анализа.
5. Если отсутствуют рост или вес — КБЖУ не рассчитывать, сначала вывести обязательный запрос этих данных.
6. Если жалобы и лекарства не указаны — вывести запрос ПЕРЕД анализом. Работать с тем что есть только по явной команде пользователя (`fallback`) или если данные переданы inline.
7. Если пользователь явно дал команду «работай с тем что есть» — **режим `fallback`**: консультация с явной пометкой об ограничениях и приблизительном КБЖУ.
8. skills/unit_conversion.md
9. skills/analyze_blood.md
10. skills/recommend.md

Выходной файл:
- clients/[Фамилия]/[ДАТА]_consult_expert.md

Вывод в чат:
- Один объединённый документ, а не два отдельных.
- Структура: 9 разделов из RULES.md, где анализ и рекомендации идут единым потоком.
- Не дублировать разделы 1-5 между анализом и рекомендациями.

### /client [Фамилия]

Клиентская версия на основе последней expert-консультации. Пайплайн: `v2/PIPELINES.md`.

Порядок:
1. RULES.md
2. Проверка обязательных данных клиента (пол, возраст, рост, вес, физическая активность).
3. Если отсутствует хотя бы один обязательный параметр и пользователь не дал явную команду "работай с тем что есть" — сначала вывести запрос недостающих данных и не запускать генерацию client-версии.
4. Если пользователь явно дал команду "работай с тем что есть" — выполнить client-версию с явной пометкой об ограничениях и приблизительном КБЖУ.
5. Последний clients/[Фамилия]/*_consult_expert.md
6. skills/recommend.md (адаптация в Client-режим)
7. Перед выводом в чат выполнить автопроверку чек-листа из RULES.md, раздел 3.2.

Выходной файл:
- clients/[Фамилия]/[ДАТА]_client_report.md

Вывод в чат:
- Один объединённый документ, не два отдельных.

Важно:
- Структура строго по RULES.md: 8 обязательных разделов (включая КБЖУ и ГДЕ КУПИТЬ).
- В клиентской версии сначала описывать, что показали анализы (без нутрицевтиков), затем переходить к блоку "что будем делать".
- КБЖУ обязателен: итоговые цифры, объяснение "почему именно столько", распределение по приёмам пищи с примерным временем и норма воды.
- Клиентская версия должна быть подробной и тёплой, без формата "огрызка".
- Минимальный объём client-версии: не менее 250-300 строк.
- В клиентской версии запрещены медицинские аббревиатуры: писать термины полностью.
- Если запрос точечный (добавить КБЖУ или ссылки), действует правило минимального вмешательства: не переписывать весь документ.
- Раздел "Где купить" с ссылками на назначенные нутрицевтики обязателен.
- Правило "без таблиц" обязательно и для client-версии.
- Если клиент принимает лекарства, раздел лекарственных взаимодействий обязателен и пишется простым языком.
- Раздел поддержки микробиома обязателен при назначении берберина или при выявленной инсулинорезистентности.

## Правило приоритетов

При конфликте инструкций приоритеты определяются по `v2/HIERARCHY.md`. Краткий порядок: эталоны → v2/ → RULES.md → skills/*.md. При конфликте v2/ с RULES.md побеждает v2/.
