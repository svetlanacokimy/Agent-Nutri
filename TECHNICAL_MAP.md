## 3) Пробелы (что не определено, но критично для стабильной генерации)

### 3.1 Нет канонического контракта profile.md
Наблюдение:
- Обязательные поля перечислены, но нет единого фиксированного формата ключей/секции/пустых значений.

Риск:
- Разные команды (/intake, /consult, inline-merge) могут записывать данные в разной структуре.

### 3.2 Нет детерминированного шаблона для итоговых разделов 6-9 в recommend.md
Наблюдение:
- Есть требования глубины, но нет жёсткого шаблона заголовков и подпунктов с проверяемым порядком.

Риск:
- Нестабильная структура между клиентами и версиями.

### 3.3 Нет единого gate-механизма "strict/fallback"
Наблюдение:
- Есть исключение «работай с тем что есть», но нет формальной модели переключения режима.

Риск:
- Конфликтные решения в зависимости от того, какой файл прочитан последним.

### 3.4 Нет явной политики anti-recursion для recommend входа
Наблюдение:
- skills/recommend.md:17 допускает вход из уже собранного consult-файла.

Риск:
- Повторное прохождение по уже агрегированному документу может закреплять дубли и искажения.

### 3.5 Нет формального quality-gate для запрета аббревиатур в client
Наблюдение:
- Правило есть (RULES.md:126), но нет явного технического чекера в пайплайне.

Риск:
- Случайные медицинские аббревиатуры могут проходить в финальный клиентский текст.

### 3.6 Нет формального правила приоритета при нескольких одновременно активных ограничениях
Наблюдение:
- Есть общий приоритет файлов (RULES.md:290-296), но нет сценарной матрицы для конфликтов типа:
  "нет данных + пользователь просит срочно + нужен точный КБЖУ".

Риск:
- Непредсказуемое поведение в граничных кейсах.

## 4) Цепочка выполнения /consult (as-is execution chain)

### 4.1 Текущий маршрут по документам
1. Прочитать RULES.md (AGENT.md:33, AGENT.md:51).
2. Проверить обязательные поля profile.md и inline-данные (AGENT.md:54-57).
3. Решить, запускать анализ или запрашивать недостающее (AGENT.md:58-60, skills/analyze_blood.md:49-60).
4. Нормализовать единицы через skills/unit_conversion.md (AGENT.md:61, skills/analyze_blood.md:64-67).
5. Выполнить аналитические этапы 0-7 через skills/analyze_blood.md.
6. Передать выводы в skills/recommend.md и сформировать тактику 6-9 (AGENT.md:63, skills/recommend.md:26-60).
7. Собрать один итоговый expert-документ из 9 разделов (AGENT.md:69-70).
8. Обновить learning/cases.md (AGENT.md:43).
9. Сохранить файл, затем commit/push и вывести полный текст в чат (AGENT.md:44-45).

### 4.2 Критические точки отказа в текущей цепочке
- Gate входных данных: конфликт strict/fallback (см. блок 2.1).
- Source policy: конфликт expert/client по отображению источников (см. блок 2.2).
- Структурная консистентность: дубли требований между RULES и skills повышают риск рассинхрона.

## 5) Критические требования к новой golden-версии skills/recommend.md

### 5.1 Границы ответственности (обязательно)
1. recommend.md владеет только разделами 6-9 expert-документа.
2. Любое содержимое разделов 1-5 считается ошибкой маршрутизации и блокируется.

### 5.2 Единый контракт входа
1. Вход: нормализованный аналитический блок + профиль клиента + список лекарств + флаг режима strict/fallback.
2. Если вход невалиден, возвращать структурированный stop-report, а не свободный текст.

### 5.3 Единый контракт выхода
1. Фиксированные заголовки и порядок для 6-9 без вариаций.
2. Для каждого нутрицевтика обязательные подполя: цель, форма, доза, тайминг, взаимодействия, ожидаемая динамика.

### 5.4 Явный режим strict/fallback
1. strict: при критически неполных данных — остановка и запрос.
2. fallback: работа с ограничениями и явной маркировкой приблизительности.
3. Режим передаётся в recommend.md явно, не определяется эвристикой.

### 5.5 Единая политика источников по режимам
1. expert-output: источники отображаются.
2. client-output: источники скрываются, но остаются во внутреннем audit-слое.

### 5.6 Нормализация глубины в одном месте
1. Единственный числовой стандарт: нутрицевтики >=25 строк, цепочки >=7 звеньев, продукты >=8 строк.
2. recommend.md не должен дублировать альтернативные диапазоны.

### 5.7 Формальная проверка лекарственных взаимодействий
1. Обязательный блок присутствует всегда.
2. При отсутствии списка лекарств выводится стандартное предупреждение строго по шаблону.

### 5.8 Формальная проверка поддержки микробиома
1. Триггеры: инсулинорезистентность и/или берберин.
2. Если триггер активен, блок обязателен и не может быть пропущен.

### 5.9 Anti-recursion защита
1. Если вход уже содержит финальную сборку consult_expert, использовать только аналитические части как read-only контекст.
2. Запрет повторной генерации на основе собственного предыдущего тактического блока без сравнения динамики.

### 5.10 Машинный self-check перед финализацией
1. Pass/fail по структуре разделов.
2. Pass/fail по глубине блоков.
3. Pass/fail по запретам (аббревиатуры в client, терапия в разделах 1-5, таблицы).
4. При fail возвращать диагностический отчёт с указанием нарушенных критериев.
